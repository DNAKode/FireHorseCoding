# Survey: Temporal & Bitemporal Data Systems

*Research-agent survey for Gneiss, 2026-07-04; statuses verified against public sources as of early July 2026 unless flagged. Requirements R1–R9 defined in [14-PRIOR-ART-MATRIX.md](14-PRIOR-ART-MATRIX.md).*

## TL;DR — top 5 takeaways for Gneiss

1. **XTDB v2 is the proof that bitemporal SQL:2011 semantics can be made ergonomic — copy its defaults, not its engine.** Query defaults of "valid-time as of now, system-time as best known" make a bitemporal store feel atemporal until you ask for history; its deliberate deviation from SQL:2011 (UPDATE applies now→∞ instead of all-of-valid-time) is the correct default and Gneiss should adopt it. But XTDB *replaces* your spine (R9 fail), so it is prior art, not a component.
2. **Datomic's reified transaction is the single best idea in this survey for Gneiss's assertion model.** Every fact is tagged with the transaction entity that asserted it, and that transaction is itself an entity you can attach facts to (source, method, operator, decision rationale). That is exactly Gneiss's "assertions about assertions" mechanism, proven in production for 14+ years — but note Datomic has **no valid time** (confirmed), so it is only half of R1.
3. **Nobody has R3 or R5.** Defeasible machine hypotheses with durable human accept/reject decisions, and explicit evaluation contexts (source precedence, closure/missingness policy, restatement policy) do not exist as first-class constructs in any surveyed system. The closest fragments: Datomic tx-annotations (R3 substrate), XTDB/Datomic as-of basis (a *time-only* evaluation context). This is Gneiss's genuinely novel territory.
4. **Marten is the strongest practical substrate on Gneiss's target stack.** Append-only Postgres event store + rebuildable async projections inside the *same* database as your conventional tables is the only surveyed system that scores well on R2, R7-coarse, R8 *and* R9 simultaneously. Gneiss's "materialized belief views" are architecturally Marten async projections, generalized.
5. **Use Snodgrass vocabulary verbatim and steal Anchor Modeling's non-destructive schema evolution.** Valid time / transaction time, current vs. sequenced vs. nonsequenced queries — this vocabulary is settled; inventing new terms costs credibility. Anchor Modeling's rule that schema evolution only ever *adds* tables (never alters) is the only surveyed pattern that makes R4 structurally cheap.

## XTDB v2 (JUXT)

**Data model.** Immutable, columnar (Apache Arrow), dynamic SQL database where *every* table is implicitly a SQL:2011 bitemporal table with system-maintained `_system_from/_system_to` and user-controllable `_valid_from/_valid_to`. Speaks a full SQL dialect over the Postgres wire protocol plus XTQL (datalog-flavored). Schemaless — nested documents, schema-on-read via Arrow types. Storage decoupled onto object stores; log + object-store architecture.

**Time axes.** Both, natively. System time is database-controlled and read-only ("users rightly have no control over system-time"). Valid time is user-settable per statement (`FOR PORTION OF VALID_TIME FROM…TO…`, `FOR VALID_TIME AS OF`, `FOR ALL VALID_TIME`). Default queries see current valid time at latest system time. Known deviation from SQL:2011: an UPDATE without a portion clause is effective now→end-of-time, not for all valid time.

**Correction/retraction.** Corrections are new writes; old system-time versions remain queryable forever. DELETE hides a row from the timeline onward; `ERASE` is the explicit escape hatch that irrevocably removes a record from disk across all valid and system time (GDPR/PIPEDA), deliberately sparingly used.

**Schema evolution.** Trivially permissive (schemaless), which means R4 is *unaddressed*: there is no ontology to version, and nothing helps you interpret old rows under the semantics they were written with.

**Maturity/licensing.** MPL-2.0. v2.0.0 GA June 2024; active 2.2.x-beta line in mid-2026 (ADBC/Flight SQL support, background GC of superseded files); JUXT reports production deployments across client sites. Younger and far less battle-tested than Postgres/SQL Server; treat as credible but early-majority.

**.NET/SQL interop.** Postgres wire protocol means Npgsql mostly works; but it is its own database — you migrate to it, you don't compose with an existing SQL Server spine.

**R-coverage.** R1 full, R2 full, R3 none, R4 none, R5 partial (bitemporal basis only), R6 partial (documents+relational, not TS/media), R7 none, R8 none (no user-defined incremental views), R9 none.

**Steal / Avoid.** *Steal:* (1) the query defaults — bitemporal store that acts atemporal unless asked; (2) the now→∞ UPDATE default; (3) `ERASE` as a named, audited, legally-motivated exception rather than pretending append-only is absolute. *Avoid:* adopting it as the store — it fails R9 and would own your data; also avoid its schemalessness, which forfeits R4.

## Datomic (Nubank)

**Data model.** Facts are datoms: `[entity attribute value transaction added?]` — EAV plus a transaction ID and an assert/retract flag, accumulating in a single immutable log. The database is a *value*: `db.asOf(t)`, `db.since(t)`, `db.history()` return stable views. Transactions are reified entities; you can transact facts *about* a transaction (who, why, source document, correction-of). Schema is itself data (attributes are entities). Peers cache and query locally; the transactor serializes writes; storage delegates to pluggable backends — including SQL databases.

**Time axes.** Transaction time only — **confirmed: no valid time**. `:db/txInstant` records when the DB learned a fact. Official guidance is to model domain/valid time as ordinary attributes; asOf/history only rewind what-was-known.

**Correction/retraction.** Retraction is an ordinary new datom (`added?=false`) — history preserved, exactly Gneiss's R2 posture. `Excision` is the separate destructive escape hatch, and the docs are explicit that excision "should never be used to correct erroneous data" — corrections are retractions/new assertions; excision exists for legal forgetting only.

**Schema evolution.** Additive schema-as-data; attributes are rarely altered, new attributes are cheap. No mechanism for interpreting old data under old schema semantics (R4 partial at best).

**Maturity/licensing.** Free of licensing fees since April 2023 (Nubank); binaries distributed under Apache-2.0 terms but **source remains proprietary** — free, not open source. Very mature (2012–), runs Nubank at enormous scale.

**.NET/SQL interop.** Weak. JVM peer model; .NET access only via REST/client-api workarounds. Ironically it can *use* SQL Server/Postgres as dumb storage — composing with SQL as a byte-store, not as a semantic spine.

**R-coverage.** R1 partial (tx time only), R2 full, R3 partial (tx-annotation substrate, no hypothesis lifecycle), R4 partial, R5 partial (asOf basis + filters), R6 none, R7 partial (every fact carries its asserting tx; nothing for derived values), R8 none, R9 none.

**Steal / Avoid.** *Steal:* (1) reified transactions as the carrier for source/method/decision metadata — this is Gneiss's assertion envelope; (2) the assert/retract flag on facts rather than row deletion; (3) the excision doctrine ("never for corrections") verbatim as policy language. *Avoid:* omitting valid time — Datomic's biggest modeling gap for operational/ERP data and the one its community works around most; avoid the peer/JVM coupling.

## SQL:2011 in mainstream engines

**SQL Server (2016 → 2025/ver17).** System-versioned temporal tables only: current table + system-managed history table, `FOR SYSTEM_TIME AS OF|BETWEEN|ALL`. Transaction time only — **confirmed; no application-time periods through SQL Server 2025** (docs still describe only system versioning; confidence high, though no explicit "not supported" statement found for 2025). History is system-managed and normally immutable, but the *current* row is corrected by mutation, and schema changes propagate to (rewrite) the history table. EF Core has first-class temporal-table mapping. R1 partial, R2 partial, R3–R8 none, R9 full (it *is* the spine). *Steal:* the "current table + history table" physical layout as one candidate materialization for belief views; EF Core integration precedent. *Avoid:* relying on it for valid time or corrections-as-data — it answers "what did the row say" not "what did we believe about the world."

**MariaDB (10.4+).** The only mainstream OSS engine with *both* axes native: system-versioned tables (`WITH SYSTEM VERSIONING`), application-time periods (`PERIOD FOR`), and documented **bitemporal tables** combining them, plus `UPDATE/DELETE … FOR PORTION OF` and `WITHOUT OVERLAPS` unique keys (10.5+). R1 full, R2 partial (system versioning records changes but app-time rows are still mutated in place), R9 full — but MariaDB is not in Gneiss's target engine set. *Steal:* its DDL surface as the reference syntax if Gneiss generates temporal DDL; it is the closest running implementation of SQL:2011's full intent. *Avoid:* nothing specific — just note bitemporal *storage* still gives you none of R3–R8.

**PostgreSQL.** PG18 (Sept 2025) shipped native application-time building blocks: temporal primary/unique keys via `WITHOUT OVERLAPS` and temporal foreign keys via `PERIOD` (GiST + btree_gist). PG19 devel docs add a "Temporal Tables" chapter and state explicitly that **system time is not supported natively** ("could be emulated using triggers; external extensions provide such functionality"); `FOR PORTION OF` not yet documented. Extensions: `periods` (xocolatl — emulates SQL:2011 periods + system versioning), `temporal_tables` (arkhipov, trigger-based, old but widely used), `pg_bitemporal` (Dombrovskaya — bitemporal conventions as functions; academic-flavored, modest adoption). R1 partial-native/full-with-extensions, R9 full. *Steal:* PG18's decision to build app-time on **range types + exclusion constraints** — ranges as first-class values, `[)` semantics, GiST enforcement is precisely the right physical vocabulary for Gneiss's valid-time columns on any Postgres deployment, and worth emulating with computed columns + filtered indexes on SQL Server. *Avoid:* waiting for native Postgres bitemporality; the trajectory (18→19) says app-time first, system-time undetermined.

## Versioned/graph databases: Fluree, TerminusDB, Dolt

**Fluree.** RDF/JSON-LD graph over an immutable, cryptographically signed commit ledger; every prior state reconstructable; SPARQL + JSON-LD query. Rewritten in Rust; v4.x, BSL-1.1 license (converts to Apache-2.0 at change date); "FlureeDB" GA relaunch June 2026 with impressive published benchmarks; production claims include DoD, Morgan Stanley, AP. Time axis: commit/transaction time only, as far as could be determined — **could not verify any valid-time support; treat as tx-time-only (moderate confidence)**. R1 partial, R2 full, R7 partial (cryptographic provenance of *commits*, not derived values), R9 none. *Steal:* signed commits as tamper-evident provenance for high-stakes evidence (industrial/compliance contexts); JSON-LD context versioning as a lightweight ontology-version pointer. *Avoid:* the "verifiable ledger" framing as a core requirement — signatures authenticate custody, they don't solve belief revision.

**TerminusDB.** Document-graph DB with git semantics: branch, merge, diff, squash over an immutable layer/commit graph; schema (a typed document layer) is versioned *in the same commit graph as the data* — the best R4 gesture in this survey, since checking out an old commit yields both old data and the schema it satisfied. Prolog/Rust core, Apache-2.0. Stewardship transferred to DFRNT during 2025; v12 released Dec 2025; alive but small — production credibility is niche. R1 partial (commit time; branches ≠ valid time), R2 full within the commit graph, R4 partial-strong, R9 none. *Steal:* (1) schema and data versioned together, so any historical read is interpretable under its contemporaneous schema; (2) branches as *what-if evaluation contexts* (a candidate mechanic for Gneiss's tentative-hypothesis workspaces). *Avoid:* betting on the ecosystem; single-vendor-small-team risk is real despite the handover.

**Dolt / Doltgres.** MySQL-compatible SQL database where every commit is a Merkle-DAG snapshot: `dolt diff` at cell level, branch/merge with conflict detection, `AS OF <commit>` queries. Dolt is mature (1.x since 2023, Apache-2.0, real production users); **Doltgres (Postgres-flavored) hit beta April 2025 and 1.0 was announced for August 6, 2026** — as of this survey it is late beta. Commit time only; no valid time; schema versioned in commits (R4 partial). R1 partial, R2 partial (history preserved *if* you commit; working set mutates freely), R9 partial (it is *a* SQL engine, but a replacement, not an overlay). *Steal:* cell-level diff between two belief states as a UX primitive — "what changed between Tuesday's report basis and today's" is a diff, and Dolt proves people will use it; merge-with-conflicts as a model for reconciling human decisions with pipeline rebuilds. *Avoid:* commit-granularity history as a substitute for per-assertion transaction time; a commit bundles unrelated changes and loses per-fact provenance.

## Event sourcing in .NET: Marten, KurrentDB, Equinox

**Marten (JasperFx).** Postgres as document DB + event store in one library: streams of JSONB events, `AggregateStreamAsync` for live rebuild, projections in three lifecycles (inline/transactional, async daemon, live), all rebuildable from events at will. Very active: v9.12.0 (June 2026), v8 line maintained; MIT; commercial support via JasperFx; pairs with Wolverine. Time axes: event append order/timestamp = transaction time; valid time only if you put it in event payloads (nobody does by default). Corrections = compensating events; GDPR = crypto-shredding patterns or stream rewrite (community guidance, not built-in). Schema evolution via event upcasters. R1 partial, R2 full, R3 none (but trivially modelable as decision events), R4 partial (upcasting ≠ faithful historical interpretation), R7 partial (projection provenance is coarse: "this view came from these streams"), R8 full (async projections are exactly incremental materialized views with principled rebuild), R9 **full** — events, documents, projections, and your conventional tables share one Postgres, one transaction. *Steal:* (1) the projection-lifecycle taxonomy (inline/async/live) for Gneiss belief views; (2) rebuild-as-a-first-class-operation with progress tracking; (3) inline projections in the same ACID transaction as the append — belief view and evidence can commit atomically. *Avoid:* event-per-aggregate-stream granularity as the *only* shape — Gneiss assertions are finer-grained than DDD aggregates; and avoid Marten's implicit "events are the whole truth" — Gneiss's evidence includes bulk re-scrapes that don't fit stream semantics well.

**KurrentDB (formerly EventStoreDB — rename confirmed, first KurrentDB release 25.0, March 2025).** Purpose-built append-only log of immutable events; streams + `$all`; persistent subscriptions; projections. License is Kurrent License v1 — source-available, enterprise features key-gated; **no longer OSI open source**. Active: 25.1 (Oct 2025: multi-stream atomic appends, secondary indexes), 26.x (2026: Kafka source connector, relational sinks to PostgreSQL/SQL Server, custom indices), 26.1 (May 2026: Arrow Flight SQL query surface with JDBC/ODBC/ADBC). Corrections = compensating events; deletion = soft delete / tombstone + scavenge, plus crypto-shredding guidance (deletion mechanics from training knowledge; high confidence). R1 partial, R2 full, R8 partial (via relational sink connectors — note they now ship exactly Gneiss's "materialized view into a SQL spine" pattern as a product feature), R9 partial (sidecar next to the spine, second operational system). *Steal:* the 26.x relational-sink direction validates Gneiss's evidence-log→SQL-belief-view topology; multi-stream atomic append shows demand for cross-entity evidence transactions. *Avoid:* the license (KLv1 restricts redistribution scenarios), and avoid a second stateful cluster when Marten gives the same semantics inside Postgres.

**Equinox (Jet/Walmart).** Not a store — a set of low-dependency F#/.NET libraries for stream-level event-sourced decision logic over pluggable backends (CosmosDB, DynamoDB, EventStoreDB/Kurrent, message-db, SqlStreamStore); Propulsion handles projections/reactions. v4.1.0 (June 2025), maintained, niche-but-respected F# community. R-coverage inherits from backend; the library itself contributes R2 discipline only. *Steal:* the access-pattern abstraction — folds + snapshots ("Transact/Query" with isolation of decision code from store) is a clean shape for Gneiss's belief-revision functions; its rigorous separation of *deciding* from *storing*. *Avoid:* nothing hazardous; just note it demonstrates the cost of backend-agnosticism — lowest-common-denominator store features.

## Bitemporal modeling literature & patterns

**Snodgrass** (*Developing Time-Oriented Database Applications in SQL*, 1999; TSQL2 work). Contributes the settled vocabulary Gneiss must use as-is: **valid time** vs **transaction time**; **current** ("what is true now as best known"), **sequenced** ("history of what is true"), and **nonsequenced** (period-agnostic) queries; state tables vs event tables; the observation that most hand-rolled temporal SQL is subtly wrong (his book is essentially an errata catalog for it). *Steal:* the current/sequenced/nonsequenced query taxonomy as the type system for Gneiss report evaluation contexts (R5) — it is the missing name for "as-known-then vs restated."

**Fowler** (patterns: Effectivity, Temporal Property, Temporal Object, Audit Log; "Bitemporal History" essay, 2021). Contributes the two-dimensional mental model (record time × actual time) in developer-accessible language, and the crucial framing that **a correction creates a new record-time branch of the actual-time history** — Gneiss's restatement policy is literally "which record-time branch do you read." *Steal:* his advice that most objects need only *some* temporal properties — full bitemporality everywhere is a tax; make it per-predicate.

**Anchor Modeling** (Rönnbäck; 6NF). Anchors (identities), attributes (one table per property, optionally historized), ties (relationships), knots (shared values); explicitly bitemporal variants. Its killer property: **schema evolution is strictly additive** — a new attribute is a new table; nothing existing is ever ALTERed, so every historical query remains valid forever (the strongest R4 answer surveyed). Cost: table explosion, view-layer dependence, weak tooling. *Steal:* additive-only evolution + "latest view" generation (their views reconstruct convenient wide rows — precursor of Gneiss materialized belief views). *Avoid:* full 6NF physically; use it as the *logical* assertion model compiled into pragmatic physical tables.

**Data Vault 2.0** (Linstedt). Hubs (business keys), links (relations), satellites (descriptive attributes, insert-only, load-date + **record-source** on every row, hash keys). Effectively transaction-time-only insert-only warehousing with provenance-lite baked into every row. *Steal:* mandatory `record_source`/`load_date` on every satellite row (Gneiss's source+method columns have 20 years of warehouse precedent here); the hub/satellite split mirroring Gneiss's entity/assertion split. *Avoid:* DV2's conflation of load time with knowledge time, and its silence on valid time and corrections semantics (corrections are just newer satellite rows — no retraction concept).

## Lakehouse time travel (Iceberg/Delta) — brief

Both give **table-level snapshot lineage**: every commit produces an immutable snapshot; query by snapshot ID/timestamp. This is a *single* time axis — a coarse transaction time for the whole table — with none of: valid time, per-fact identity across versions, correction semantics (a rewritten file silently replaces facts), or durable history (snapshot expiry/`VACUUM` routinely garbage-collects the past for cost). Snapshot time travel answers "what did this table contain when job X ran" (useful for reproducibility of pipelines), not "what did we believe about entity E and when." R1 partial-tx-only, R2 none (history is disposable by design), rest none. *Steal:* the snapshot-pin for pipeline reproducibility — a Gneiss evaluation context should be able to pin upstream extract snapshots. *Avoid:* ever calling snapshot time travel "bitemporal" in Gneiss docs; and note the retention lesson — history that costs money gets deleted unless it is semantically load-bearing.

## Gaps none of these fill

- **R3 (defeasible hypotheses + human adjudication as durable first-class data): universal gap.** No surveyed system models "machine proposed link L with confidence c; human H rejected it on date D; that rejection must survive a pipeline rebuild." Fragments exist — Datomic tx-annotations could *store* it, TerminusDB branches could *stage* it, Marten could *event* it — but nothing gives the hypothesis lifecycle (proposed → confirmed/rejected → superseded) schema, semantics, or rebuild-survival guarantees. This is Gneiss's clearest novel contribution.
- **R5 (evaluation contexts beyond time): universal gap.** Every system with any time travel offers a *temporal* basis (as-of). None lets a query declare source precedence, closure/missingness policy (closed-world per-source? open?), or restatement policy as a named, versioned, auditable object. Snodgrass's query taxonomy names a third of it; nobody implements the rest.
- **R6 (per-modality storage under one semantic layer): universal gap in this topic area.** Every surveyed system is one-engine-fits-all (XTDB, Datomic, Fluree) or one-modality (Kurrent). Marten's document+event+relational-in-one-Postgres is the nearest gesture but doesn't reach time series or media.
- **R7 fine-grained justification: universal gap.** Event sourcing gives coarse lineage (view ← streams); nothing records *why this cell has this value* (which assertions won, under which precedence rule). Provenance-of-derivation exists in academic work (PROV, provenance semirings), not in any surveyed production system.
- **R4 with faithful historical interpretation: near-universal.** Only TerminusDB/Dolt version schema *with* data, and even they can't evaluate one query under two ontology versions to explain a restatement. Anchor Modeling sidesteps rather than solves it (never change anything, only add).
- **R1+R9 together: the composition gap that motivates Gneiss.** Full bitemporality exists only in systems that replace your database (XTDB, MariaDB-if-you-switch). Everything that composes with an existing SQL spine (SQL Server temporal, Marten, extensions) delivers at most one time axis. No surveyed system offers bitemporal, append-only, provenance-carrying semantics *as a layer over* SQL Server/SQLite/Postgres — which is precisely the slot Gneiss targets.
